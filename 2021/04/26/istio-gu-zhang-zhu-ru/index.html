1. 如何进行故障注入

1. 故障从而何来

下载了istio整个文件夹后，在根目录能看到一个samples文件夹，samples文件夹下有bookinfo文件夹，该文件夹内包含了所有bookinfo相关的内容。再往下，bookinfo文件中可以看到networking文件夹，这个文件夹里有所有对bookinfo进行网络控制的yaml配置文件。

上图展示了networking文件夹的一些yaml配置文件，其中故障相关的网络配置文件是：virtual-service-ratings-test-abort.yaml, virtual-service-ratings-test-delay.yaml

同时，我们进行故障注入时也是围绕这两个配置文件进行，所谓故障不过是一种病态的流量模式。

2. 如何自定义我们需要的故障

出于个人需求，已经将istio官方提供的这两个文件修改了一些匹配规则，这里也只展示个人修改结果：

apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: reviews-route
  namespace: default
spec:
  hosts:
    - reviews
  http:
  - match:
    - uri:  # uri的匹配规则还可以根据用户进行，只需前往istio官网学习相关匹配规则即可
        regex: ".*reviews.*"  # 这里表示匹配所有对reviews的请求，因为不清楚微服务内部各app之间具体的请求URL故此用通配符
    fault:
      delay:  # 这里表明了故障类型， istio提供了delay和abort
        percentage:
          value: 25.0   # 这里表示故障概率，0.0~100.
        fixedDelay: 2s  # 这里表示时延的长短，在abort中断故障中无此字段
    route:
    - destination:
        host: reviews
        subset: v2
  - route:
    - destination:
        host: reviews
        subset: v2

3. 如何根据配置文件运用规则

# 运用相应规则
kubectl apply -f <config>.yaml

# 删除相应规则
kubectl delete -f <config>.yaml

2. 更改故障注入对象：

先给出参考的网站链接:

https://istio.io/latest/docs/tasks/traffic-management/

1. 相应的规则只对配对用户发出的请求生效

http:
  - match:
    - headers:
        end-user:
          exact: jason

2. 相应的规则只对匹配的URI对应的请求生效

http:
  - match:
    - uri:
        prefix: /reviews  
        # 上面我进行URI匹配时不写为这样是发现
        # 人为直接对reviews进行请求时,可以用uri /reviews进行请求,但是微服务内部各自相互请求时,并非用该URI
        # 如果使用该URI进行配置,故障规则将不会发生
